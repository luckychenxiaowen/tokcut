# 一个下午，用 AI 写了个开源工具，把国产大模型的 Token 开销砍了 40%

我是那种会被一个小 demo 搞得睡不着的程序员。

上周在 GitHub 上偶然看到一个项目叫 Caveman，思路极其简单——往 system prompt 里塞一段强制简洁输出的指令文本，什么都没改，就在 Claude Code 上平均省了 65% 的输出 token。

65%。我反复看了三遍这个数字。

然后我开始想：如果把这个思路扩展成输入压缩 + 输出压缩 + 语义缓存三件套，做成一个透明代理，覆盖国产主流大模型（DeepSeek、GLM、混元、Kimi、MiniMax），能省多少？

说干就干。一个下午加一个晚上，用 DeepSeek + CodeBuddy 搭出了一个叫 tokcut 的工具，在五个模型上跑了一遍：平均**省下来 41% 的 token**。代码已经开源在 GitHub 上了。

下面聊聊过程和一些思考。

---

## 为什么 token 这么难省？

嘴上说「让模型少废话」当然容易。真做起来有三道坎，一道比一道恶心：

**第一道坎：废话怎么定义？**  
用户问「请问一下，能不能麻烦你帮我看看这段代码的问题」，这里面的「请问」「麻烦」「能不能帮我看看」算废话吗？在英文里 maybe 是，但中文语境下删了这些词，模型是有可能理解偏的。可你要不删，它们就是实打实吃 token。

**第二道坎：压缩怎么不丢信息？**  
如果在压缩文本的过程中，把代码块里的内容给截断了，把 URL 里的路径给丢了，把版本号 `3.10.1` 压缩成 `3.10`，那省下来的 token 毫无意义。

**第三道坎：你怎么处理重复请求？**  
这是最容易被忽略的。很多场景下用户的提问高度相似。比如「帮我写一个 Python 的冒泡排序」和「用 Python 实现冒泡排序」，在 token 层面几乎等价，但如果不做缓存，每次都会重新走一遍 LLM。

这三道坎，就是 tokcut 要解决的核心问题。

---

## 怎么挑技术方案

动手之前，我把能搜到的 token 节省方法都扒了一遍。真有十几种——prompt 压缩、语义缓存、KV cache 压缩、本地路由、CoT 压缩、diff 编辑……各有各的坑。

挑了三个标准筛：

1. **不绑特定模型**。CoT 压缩和 KV Cache 压缩听着高级，但它们吃的是模型推理时的内部状态，换一个模型就废了。我要的是通用工具，这种绑模型的直接 pass。

2. **复杂度别太高**。本地路由理论上很美——本地小模型先判断要不要调云端大模型。但判断准确率是硬伤，误判一次，没省到反而多花了。hold 不住的东西不碰。

3. **零侵入**。用户感受就是「base_url 换了一下」，然后该干嘛干嘛。但凡要求别人改代码的，都算输。

最终留下的三个模块，恰好覆盖了请求生命周期的三个关键节点：

| 节点 | 模块 | 核心策略 |
|------|------|---------|
| 请求到达时 | **语义缓存** | 用 sentence-transformers 把请求转成向量，和历史缓存比余弦相似度，超过 0.95 直接返回 |
| 转发前 | **输入压缩** | safe 模式去重行 + 去填充词，aggressive 模式额外截断 30% |
| 返回时 | **输出压缩** | 在 system prompt 里注入 Caveman 风格指令 + 响应后处理规则兜底 |

这三个模块叠起来，在 25 个测试用例上平均省了 41%。

---

## 一些具体的技术选择

**为什么用 sentence-transformers 而不是直接用文本哈希做缓存？**

文本哈希的问题在于太严格了。用户说「帮我解释 useEffect」和「useEffect 是什么」，哈希值完全不同，但语义上几乎一样。如果你的缓存命中率不到 10%，那这个缓存层就是摆设。

选 `all-MiniLM-L6-v2` 是因为它在速度和精度之间是个很甜的平衡点。80MB 的模型，在 CPU 上做一次编码大概几十毫秒，对请求延迟的影响可以忽略。

**为什么 ContentProtector 要放两遍？**

输出压缩的第一步是向 model 注入指令，第二步是后处理响应文本。这两步都需要保护技术内容。

在注入阶段，你当然不想让 system prompt 里的保护指令被后处理吃掉。在处理响应文本的时候，你又不想把代码片段也「优化」掉。所以 protector 在两个环节各执行一次 protect-restore 循环：注入的时候只是追加指令，后处理的时候才是真正的规则压缩。

**动态控制的设计动机**

我不喜欢「改配置文件然后重启服务」这种操作。真实使用场景里，你可能前一个请求想用 ultra 模式极简输出，下一个请求又想关掉压缩看看完整回答。

所以从一开始就设计成了 Header 级别的开关：

```
X-Tokcut-Level: lite|full|ultra
X-Tokcut-Compress: true|false
X-Tokcut-Cache: true|false
```

同一个服务进程，不同请求带不同 Header，互不干扰。

## 时间都花在哪了

有人问我 AI 写代码到底能快多少。我翻了翻聊天记录，大概是这样：

- **前一个小时**：读 Caveman 源码，把市面上所有 token 节省方案过了一遍，写设计文档
- **接下来一个半小时**：CodeBuddy 按设计文档一口气吐出来 33 个文件——源码、配置、Docker、CI/CD，全套
- **后面的两个小时**：调 bug、跑测试、改参数，来回好几轮。code review 这事 AI 替不了
- **最后俩小时**：补文档。中英文双语的 README、CONTRIBUTING、SECURITY……比写代码还累
- **收尾**：推到 GitHub，修了一个 gitignore 误拦文件的低级 bug

六个多小时，从零到在 GitHub 上能跑的东西。

平心而论，AI 最大的帮助是「从设计文档到代码」那一步。但设计文档本身——哪个模块该拆、接口怎么定、数据怎么流——这些还是得自己想。这部分目前 AI 帮不了忙，也不该帮。
